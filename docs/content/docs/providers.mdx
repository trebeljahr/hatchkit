Hatchkit talks to a handful of external systems. Every one is optional until a command actually needs it — you can start with just GitHub + Coolify and add the rest later.

## Credential storage

| Where | What |
|---|---|
| OS keychain (macOS Keychain / libsecret) | All tokens + secrets, scoped under the `hatchkit` service. |
| Default config file | Non-sensitive metadata — URLs, user IDs, region hints, provider status. Run `hatchkit status --json` to see `configPath`; override the directory with `HATCHKIT_CONF_DIR`. |

Re-check everything:

```bash
hatchkit config   # status
hatchkit doctor   # live API check
```

## Config directory override

`HATCHKIT_CONF_DIR` points Hatchkit at a different config directory. When it is
set, Hatchkit reads and writes `$HATCHKIT_CONF_DIR/config.json`; cached
provisioned env blocks stay beside it under
`$HATCHKIT_CONF_DIR/provisioned/<project>.{dev,prod}.env`. Secrets do not move:
provider tokens and private keys still live in the OS keychain under the
`hatchkit` service.

Use it for isolated test state, CI jobs, demos, or separate local workspaces
where you do not want commands to read or update your normal provider metadata
and assigned-port registry.

Default: unset. Without an override, Hatchkit uses the OS config directory from
the `conf` package for project `hatchkit`: on macOS,
`~/Library/Preferences/hatchkit-nodejs/config.json`; on Linux,
`$XDG_CONFIG_HOME/hatchkit-nodejs/config.json` or
`~/.config/hatchkit-nodejs/config.json`; on Windows,
`%APPDATA%\hatchkit-nodejs\Config\config.json`.

Example:

```bash
export HATCHKIT_CONF_DIR="$PWD/.hatchkit-conf"
hatchkit status --json
hatchkit setup
```

## GitHub

**Auth:** `gh auth login`. Hatchkit shells out to the `gh` CLI — no tokens of its own.

**Used by:** `create` (repo provisioning, first push).

---

## Coolify

Self-hosted PaaS that runs the final app container. Hatchkit uses the Coolify API to:

- Create the app from your GitHub repo,
- Push encrypted env vars + the dotenvx private key,
- Trigger deploys.

**You need:** a Coolify URL (e.g. `https://coolify.example.com`) and an API token with write access to projects / applications / env vars.

Configure: `hatchkit config add coolify`.

---

## Hetzner Cloud

Optional. Used when you pick **"new VPS"** as a deploy target.

**You need:** a Hetzner Cloud API token.

Hatchkit will pick a server size interactively (CX22 / CX32 / CX42 / CX52 or the CAX ARM line) and Terraform will provision the VM, attach SSH keys, and bootstrap Coolify.

Configure: `hatchkit config add hetzner`.

## DNS

Hatchkit creates the apex + `api.` records pointing at the deploy target.

**Supported:**

- **INWX** — username + password + optional 2FA shared secret
- **Cloudflare** — API token scoped to the zone

Configure: `hatchkit config add dns`.

## Email — Resend

Hatchkit can create a **restricted** Resend API key per environment (dev/prod) tied to a sending domain you own. It writes `RESEND_API_KEY` and, when the sending domain is known, `RESEND_FROM_EMAIL` for the starter email runtime.

Configure: `hatchkit config add resend`.

## Email — Listmonk + AWS SES

The sovereign alternative to Resend. Hatchkit verifies a per-project sending subdomain on SES (`mail.<projectDomain>`), publishes the 3 DKIM CNAMEs into Cloudflare, then provisions a `<project>` (live) + `<project>-test` (dev) list in your Listmonk instance. SES handles delivery + IP reputation (~€0.10 per 1,000 sends); Listmonk owns subscribe/confirm/broadcast UI. The runtime gets a quartet of env vars covering both routes (`LISTMONK_*` for the API, `SES_SMTP_*` for direct SMTP, `SES_FROM_EMAIL` for the from-address).

Configure: `hatchkit config add ses` + `hatchkit config add listmonk` (add `--deploy` to the second one to get a guided walkthrough for the Coolify Listmonk service template).

See [Email — Listmonk + SES](/docs/email) for the full setup walkthrough and the unavoidable human-in-the-loop steps (SES production-access lift, Listmonk admin wizard, SMTP relay paste).

## Search — Google Search Console

Hatchkit uses Google OAuth once, then `hatchkit add <project> search-console`
requests a Site Verification DNS TXT token, writes it through Cloudflare DNS,
verifies ownership, and adds the `sc-domain:<domain>` Search Console property.
Google does not support API-key auth for this flow; the stored credential is an
OAuth refresh token in the OS keychain. Hatchkit requests Search Console
read/write plus Site Verification verify-only access; the token stays local to
the user's machine.

Hatchkit's CLI uses Google's desktop loopback redirect flow
(`http://127.0.0.1:<port>/oauth/google/callback`). For that flow, use a Google
OAuth client with application type **Desktop app** and copy both the client ID
and the generated client secret. Hatchkit still sends PKCE (`code_challenge`
and `code_verifier`), but Google Desktop loopback clients require the generated
secret at the token endpoint in practice. Do not use a placeholder or bogus
secret; Google validates the real value. Hatchkit stores the client ID, client
secret, and refresh token in the OS keychain.

If an older stored Search Console token was created without the Desktop client
secret, `hatchkit doctor` reports `client_secret is missing` and `hatchkit add`
re-runs Search Console setup before continuing.

Maintainer/dev builds can provide a Hatchkit-owned OAuth client by exporting
both `HATCHKIT_GOOGLE_SEARCH_CONSOLE_CLIENT_ID` and
`HATCHKIT_GOOGLE_SEARCH_CONSOLE_CLIENT_SECRET` before running setup. Keep the
secret in a local shell secret manager or private env file, not in git history,
docs, generated `dist/`, npm package contents, or committed dotenvx files.

Maintainer setup:

1. Create a Hatchkit-owned Google Cloud project.
2. Enable **Google Search Console API** and **Google Site Verification API**.
3. Configure Google Auth Platform branding/audience/data access.
4. Add scopes `https://www.googleapis.com/auth/webmasters` and
   `https://www.googleapis.com/auth/siteverification.verify_only`.
5. Create an OAuth client with application type **Desktop app**.
6. For local maintainer testing, keep the app in **Testing** and add your Google
   account under **Audience → Test users**. The Verification Center will say
   verification is not required in this mode; that is expected. Testing-mode
   refresh tokens expire after seven days.
7. For a broad/public Hatchkit release, publish the app to production and submit
   the Search Console + Site Verification scopes for verification.
8. For local maintainer testing, export both
   `HATCHKIT_GOOGLE_SEARCH_CONSOLE_CLIENT_ID` and
   `HATCHKIT_GOOGLE_SEARCH_CONSOLE_CLIENT_SECRET`. If Google no longer reveals
   an existing secret, add or rotate a client secret and update only your local
   secret source.

Configure: `hatchkit config add search-console`.

## Web analytics — Plausible

Uses the **Plausible Sites API** to create a project site for the app domain and wire the browser tracker env (`NEXT_PUBLIC_PLAUSIBLE_DOMAIN` + script URL).

Plausible Community Edition/self-hosted does not include the Sites API. For CE, create the site manually in Plausible; `hatchkit add <project> plausible` can still write `NEXT_PUBLIC_PLAUSIBLE_DOMAIN` + `NEXT_PUBLIC_PLAUSIBLE_SCRIPT_URL`, but it cannot create or verify the Plausible site for you.

Configure: `hatchkit config add plausible`.
