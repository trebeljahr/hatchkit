Hatchkit has two distinct rotation flows. They live next to each other and sound similar, but they touch different things and you reach for them in different situations.

## Two kinds of rotation

### dotenvx keypair rotation — `hatchkit keys rotate`

Re-encrypts `.env.production` with a new `DOTENV_PRIVATE_KEY_PRODUCTION`. Under the hood: `dotenvx rotate -f .env.production`, prune `.env.keys` back to the new key, update the macOS Keychain / libsecret entry, then push the new private key to Coolify and the GitHub Actions secret store. The **provider credentials inside the env file do not change** — every value is decrypted with the old key and re-encrypted with the new key. This is the right command when the encryption key itself is compromised (lost laptop, leaked `.env.keys`, suspected keychain dump) or when you want a scheduled rotation of the at-rest encryption layer.

### Provider credential rotation — `hatchkit secrets rotate`

Generates new third-party credentials (a new OpenPanel write client, a new GlitchTip DSN, etc.) by talking to each upstream provider's API, and writes them into the encrypted env file. The **dotenvx keypair is unchanged** — the file is still encrypted with the same `DOTENV_PRIVATE_KEY_PRODUCTION`. Each adapter runs in two phases (mint new → verify → revoke old), so a verify failure leaves the old credential live and stashes a rollback blob in the keychain. This is the right command when a specific upstream credential is compromised (an OpenPanel secret leaked into a log, a GlitchTip DSN posted in a screenshot) or as part of scheduled per-provider rotation.

The two flows are independent and compose cleanly — if both the keypair and the upstream credentials need rotating (e.g. an ex-employee had repo access), run `hatchkit keys rotate <project>` first, then `hatchkit secrets rotate <project>`.

## When to use which

| Scenario | `keys rotate` | `secrets rotate` |
|---|---|---|
| `.env.keys` accidentally committed to git | yes | yes (after keys) |
| Laptop with keychain access lost / stolen | yes | yes (after keys) |
| Single upstream provider key leaked in a log | no | yes (`--providers=<that-one>`) |
| Ex-team-member offboarding | yes | yes |
| Scheduled at-rest encryption rotation (quarterly) | yes | no |
| Scheduled provider key rotation (quarterly) | no | yes |
| Suspected upstream provider breach | no | yes |
| Migrating dotenvx versions | yes | no |

For partial-revoke cases (only one provider compromised), `hatchkit secrets rotate <project> --providers=<name>` swaps only that adapter and leaves the rest alone. For partial deploy-target pushes (e.g. rotate locally but don't touch Coolify yet), use `--push-targets=none` or `--push-targets=gh`.

## `hatchkit secrets rotate` — command reference

```bash
hatchkit secrets rotate <project> [flags]
```

Run from anywhere — `<project>` is the project slug (matches `.hatchkit.json#name`), and the orchestrator resolves the project directory from the current working directory.

### Flags

| Flag | Default | Effect |
|---|---|---|
| `--env <scope>` | `production` | Scope the rotation. Only `production` is supported today. |
| `--providers <list>` | `all` | Comma list of adapter names, or `all`. Validated against the live registry — unknown names error cleanly. |
| `--push-targets <list>` | `both` | Comma list of `coolify`, `gh` (alias `github`), or `both` / `none`. Silently filters to whatever is actually configured + detected. |
| `--revoke-old <policy>` | `after-verify` | `after-verify`: revoke old credential only after verify succeeds. `never`: leave old credential live (safe for audit replays). `immediate`: revoke before verify (for emergency leak-race rotations). |
| `--push-gh <owner/repo>` | auto-detected | Override the GitHub repo for the Actions secret push. Alias: `--repo`. |
| `--dry-run` | off | Print the plan (providers, env-key names, deploy targets, revoke policy) without minting, writing, or pushing. |
| `--json` | off | Emit one NDJSON line of `RotationAudit` on stdout. Names + outcomes only — never credential values. |

### Examples

Rotate every detected provider, push everywhere previously configured:

```bash
hatchkit secrets rotate raptor-runner
```

Preview the plan without mutating anything:

```bash
hatchkit secrets rotate raptor-runner --dry-run
```

Rotate only OpenPanel (e.g. a leaked write-client secret):

```bash
hatchkit secrets rotate raptor-runner --providers=openpanel
```

Rotate locally only — no Coolify or GitHub push (you intend to handle the fan-out yourself):

```bash
hatchkit secrets rotate raptor-runner --push-targets=none
```

Emergency leak race — revoke the old credential before verifying the new one. Use when you are racing an exfiltrated key and accept the small window where neither old nor new may work:

```bash
hatchkit secrets rotate raptor-runner --revoke-old=immediate
```

Audit-friendly rotation — mint the new credential, push it, verify it, but leave the old one live for a later manual revoke (useful when you need a window to update consumers that pin the old key):

```bash
hatchkit secrets rotate raptor-runner --revoke-old=never
```

Machine-readable output for CI:

```bash
hatchkit secrets rotate raptor-runner --json | jq '.adapters[] | {provider, verificationResult, oldRevoked}'
```

## Supported providers

The registry today:

| Adapter name | Label | Env keys rotated | Verify probe | Notes |
|---|---|---|---|---|
| `openpanel` | OpenPanel write client | `OPENPANEL_CLIENT_ID`, `OPENPANEL_CLIENT_SECRET`, `PUBLIC_OPENPANEL_CLIENT_ID` | `GET /manage/projects` with the new client | Additive rotation — the new client exists alongside the old until revoke runs. |
| `glitchtip` | GlitchTip DSN | `GLITCHTIP_DSN`, `PUBLIC_GLITCHTIP_DSN` | `POST /api/<project-id>/store/` with a minimal Sentry envelope | Split-surface projects rotate two distinct DSNs (one per GlitchTip project: `<name>-server` and `<name>-client`). Non-split surfaces share a single DSN across both env vars. |

Run `hatchkit secrets rotate <project> --dry-run` to see which adapters detect against your project — `envPresence` (the union of names found in `.env.development` plain and `.env.production` decrypted) is the canonical signal each adapter consults.

## Adding a new provider adapter

Adapters live one-per-file under `cli/src/secrets/adapters/`. They self-register at import time via `register(adapter)` from `../registry.js`. The orchestrator iterates the registry — adding a new provider is "drop a file, import it from the barrel, done". No switch statements, no central manifest.

### The `ProviderRotator` interface

```ts
// cli/src/secrets/types.ts
interface ProviderRotator {
  readonly name: string;   // stable slug used in --providers=<name>
  readonly label: string;  // human-readable label for log output

  // Pure / read-only. Returns true when this adapter should run.
  // Combines ctx.envPresence with manifest hints.
  detect(ctx: RotationContext): boolean;

  // The env-var names this adapter owns. Returned upfront so dry-run
  // can print them and the orchestrator can collision-check.
  envKeys(ctx: RotationContext): ReadonlyArray<EnvKeySpec>;

  // Read the CURRENT credential so the orchestrator can stash it in
  // the rollback store BEFORE any mutation. Empty values map → the
  // orchestrator downgrades revoke policy to 'never' for this adapter.
  captureOld(ctx: RotationContext): Promise<OldCred>;

  // Mint a fresh credential from the upstream provider. Side-effect-
  // only against the upstream API — do NOT write to disk, keychain,
  // or deploy targets. The orchestrator owns those steps.
  createNew(ctx: RotationContext): Promise<NewCred>;

  // Smoke-test the NEW credential. Return 'skipped' when no cheap
  // probe exists — the orchestrator will not revoke the old key.
  verify(ctx: RotationContext, fresh: NewCred): Promise<VerifyOutcome>;

  // Revoke the OLD credential upstream. Must be idempotent — a 404
  // is success.
  revoke(ctx: RotationContext, old: OldCred): Promise<void>;
}
```

### Self-registration pattern

At the bottom of each adapter file:

```ts
// cli/src/secrets/adapters/<name>.ts
register({
  name: "<slug>",
  label: "<Human label>",
  detect, envKeys, captureOld, createNew, verify, revoke,
});
```

Then add one line to the barrel so the side-effectful `register` call runs:

```ts
// cli/src/secrets/adapters/index.ts
import "./glitchtip.js";
import "./openpanel.js";
import "./<new-adapter>.js";   // ← add this
```

No orchestrator changes required. Duplicate `name` collisions throw at startup so accidental shadowing is loud.

### Reference example

[`cli/src/secrets/adapters/openpanel.ts`](https://github.com/trebeljahr/hatchkit/blob/main/cli/src/secrets/adapters/openpanel.ts) is the canonical reference. It demonstrates:

- Reading from the adapter-private keychain cache populated by the matching `provision/openpanel.ts` flow.
- Stashing API-call intermediates on `ctx.scratch[ADAPTER_NAME]` so `captureOld` → `createNew` → `verify` → `revoke` don't refetch config.
- Filtering `ENV_KEYS` to whatever is actually present in the project (avoid writing `PUBLIC_*` mirrors into a backend-only repo).
- Additive rotation: mint a new client, verify it, then `DELETE` the old client by id — the new credential exists alongside the old one until revoke runs, which is what makes the rotation crash-safe.

[`cli/src/secrets/adapters/glitchtip.ts`](https://github.com/trebeljahr/hatchkit/blob/main/cli/src/secrets/adapters/glitchtip.ts) shows the same pattern with two extra wrinkles: a verify probe against the upstream's data ingest endpoint (not just a control-plane probe), and a per-env-var handle shape (`<field>:<envKey>`) for split-surface projects that own two GlitchTip projects.

## Safety notes

### dotenvx-encryption requirement

`secrets rotate` writes new credential values into `.env.production` via the canonical `writeProdEnv` helper, which encrypts every value with the project's `DOTENV_PRIVATE_KEY_PRODUCTION`. If `.env.production` is not yet dotenvx-encrypted, the orchestrator prints a yellow warning and proceeds — the rotation will encrypt it going forward, but past git commits may contain plaintext credentials. Run `hatchkit adopt` first to migrate an existing project onto dotenvx.

### `.env.keys` gitignore requirement

The orchestrator **refuses** to run when `.env.keys` is tracked by git. The reasoning is that rotating provider credentials while the dotenvx private key still sits in your git history is theatre — the encrypted `.env.production` is already readable to anyone with repo access. The error message spells out the recovery recipe:

```bash
git rm --cached .env.keys
echo .env.keys >> .gitignore
hatchkit keys rotate <project>
# then retry:
hatchkit secrets rotate <project>
```

### Audit redaction guarantee

The audit shape (`RotationAudit`) carries names + outcomes only — never credential values. Defence-in-depth runs on top: a `redactValues` walker scans every audit object before stdout and replaces strings matching credential heuristics (hex 32+ chars, common prefixes like `sk_` / `whsec_` / `ghp_`, DSN-shaped URLs, anything keyed under `*_SECRET` / `*_KEY` / `*_TOKEN` / `*_PASSWORD` / `*_DSN` / `*_PRIVATE_KEY`) with `[REDACTED]`. Adapter errors are wrapped through `redactErrorMessage` before reaching stderr for the same reason. `--json` output is safe to pipe into logs.

### Two-phase rotation guarantee

Each adapter runs `captureOld → saveRollback → createNew → write envs → push targets → verify → revoke (only when verify succeeded)`. A verify failure leaves the **old credential live** and stashes a rollback blob in the keychain under account `secrets-rollback:<project>:<adapter>`. The orchestrator clears the rollback blob only on full success (`verify === 'ok'` AND `oldRevoked === true`). Anything else keeps the blob around for operator inspection.

Adapters that cannot recover the old credential (empty `captureOld` result) silently downgrade revoke policy to `never` for that adapter — the audit picks this up via `revoke-held` and the old credential is not touched.

`--revoke-old=immediate` reverses the order (revoke first, verify second) for emergency leak-race scenarios. The operator accepts the small window where neither the old nor the new credential may work, in exchange for not leaving the old key live for the duration of the verify probe.
